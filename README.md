# SupportBot

An e-commerce customer support agent built in phases to demonstrate how AI support systems work under the hood — knowledge retrieval, order lookup, session memory, escalation, and model-driven tool orchestration. Each phase is a working demo, not a toy.

**Stack:** Python, FastAPI, Claude API (Haiku), Voyage AI embeddings, Supabase/Postgres + pgvector

## What exists today

### Agent
- `backend/app/agent.py` — model-driven tool loop using Claude API; four formal tool schemas; deterministic fallback when no API key is set
- `backend/app/main.py` — FastAPI entrypoint with `/health`, `/api/chat`, `/api/chat/stream` (SSE), `/api/compare`, and `/eval`

### Retrieval
- `backend/app/repository.py` — storage boundary; hybrid search (30% full-text + 70% cosine similarity via Voyage AI) with optional Voyage reranking and category pre-filtering; falls back to full-text, then in-memory keyword search
- `backend/knowledge/` — source markdown files for persisted support knowledge
- `backend/app/data.py` — fallback sample orders and knowledge base (no external deps needed)

### Data
- `backend/app/data_loader.py` — Olist dataset normalization, Voyage embedding generation, fixed and semantic chunking, near-duplicate dedup
- `backend/app/cli.py` — summarize, export, import-orders, import-knowledge, compare-retrieval commands
- `backend/sql/schema.sql` — Supabase/Postgres schema (orders, knowledge_documents, knowledge_chunks + pgvector)
- `backend/sql/migrate_5c_metadata.sql`, `backend/sql/migrate_7_customer_memory.sql` — additive migrations

### Evaluation
- `backend/eval/queries.json` — 30 labeled queries across refund, shipping, product, order, multi-intent, off-topic, and unanswerable categories
- `backend/eval/agent_fixtures.json` — 12 multi-turn agent fixtures
- `backend/eval/run.py` — runner with `--mode`, `--all-modes`, `--chunking-audit`, `--llm-judge`, `--benchmark`, `--agent-eval` flags; per-mode JSON in `backend/eval/results/`
- `backend/eval/check_regression.py` + `thresholds.json` — CI regression gate against committed baseline
- `backend/app/eval_dashboard.py` — `GET /eval` color-coded summary table, latency chart, per-query drill-down
- `docs/benchmark.md` — committed mode × metric table regenerated by `--benchmark`

### Frontend
- `frontend/index.html` — single-file Chat UI (SSE streaming, tool cards, source chips) and Compare tab (phase1/2/3 side-by-side)

### Customer memory (data layer)
- `backend/app/conversation_store.py` — durable conversation history protocol + in-memory implementation
- `backend/app/customer_store.py` — customer identity, memory facts, and order linkage protocol + in-memory implementation
- `backend/app/memory_context.py` — system-prompt injection helpers
- Postgres tables exist via the Phase 7 migration; the agent does not consume any of this yet

### Config & tests
- `backend/app/config.py` — reads `.env`; `SUPPORTBOT_DATA_BACKEND`, `DATABASE_URL`, `VOYAGE_API_KEY`, `ANTHROPIC_API_KEY`
- `backend/tests/` — unit tests (no external deps) + integration tests gated by env vars
- `.github/workflows/test.yml` — runs unit tests on every push and PR
- `.github/workflows/eval.yml` — runs the eval and regression gate on every PR
- `plans/supportbot-plan.md` — phased implementation plan
- `plans/decisions/chunking.md` — chunking strategy decision
- `docs/improvement-log.md`, `docs/release-roadmap.md` — release history

## Current capabilities

| Capability | Status |
|---|---|
| Product questions from knowledge base | ✅ Hybrid semantic + keyword retrieval, optional rerank + category filter |
| Order status lookup | ✅ Supabase (10k Olist rows) + in-memory fallback |
| Refund flow with delivery validation | ✅ Model-driven, multi-step |
| Escalation to support ticket | ✅ Conversational, contextual ticket subject |
| Session memory across turns | ✅ Full conversation context passed to model (in-process; lost on restart) |
| Multi-step tool chains in one message | ✅ Claude chains tools automatically |
| Confidence-based fallback | ✅ 0.25 threshold; escalates rather than guessing |
| Streaming chat UI | ✅ SSE tokens + live tool activity cards |
| Source citations on every reply | ✅ Title + score; sub-0.15 sources suppressed |
| Eval dashboard | ✅ `GET /eval` — mode × metric table, latency chart, per-query drill-down |
| Committed benchmark + CI regression gate | ✅ `docs/benchmark.md`, fails CI on >10% metric drop |
| LLM-as-judge answer correctness | ✅ `--llm-judge` flag on the eval runner |
| Cross-session customer memory | 🟡 Data layer + tests only; not wired into the agent |

## Local run (no external deps)

Install dependencies:

```bash
pip install -r backend/requirements.txt
```

Run the API:

```bash
uvicorn backend.app.main:app --reload
```

Run tests:

```bash
python3.11 -m pytest backend/tests/ -v --tb=short
```

Integration tests self-skip when `DATABASE_URL` or `VOYAGE_API_KEY` is absent, so the same command works locally and in CI.

Without `.env` set, the bot uses in-memory data and deterministic routing — no API keys required.

## Full setup (Supabase + Voyage AI + Claude)

Copy and fill in `.env`:

```
SUPPORTBOT_DATA_BACKEND=postgres
DATABASE_URL=<supabase-postgres-url>
VOYAGE_API_KEY=<voyage-api-key>
ANTHROPIC_API_KEY=<anthropic-api-key>
```

Apply the schema, then import data:

```bash
# Apply schema in Supabase SQL editor
# backend/sql/schema.sql

# Import orders from Olist dataset
python3.11 -m backend.app.cli import-orders --dataset-dir ./data-set --limit 10000

# Import and embed knowledge docs
python3.11 -m backend.app.cli import-knowledge --knowledge-dir ./backend/knowledge
```

Run full test suite (hits Supabase + Voyage):

```bash
python3.11 -m pytest backend/tests/ -v
```

Compare keyword vs hybrid retrieval:

```bash
python3.11 -m backend.app.cli compare-retrieval
```

## Example requests

Product question (semantic retrieval):
```bash
curl -s -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"session_id":"demo","message":"Can I get my money back?"}' | jq
```

Multi-step refund in one message:
```bash
curl -s -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"session_id":"demo","message":"My order ORD-1002 arrived damaged, I want a refund."}' | jq
```

Real Olist order lookup:
```bash
curl -s -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"session_id":"demo","message":"Where is order 000229ec398224ef6ca0657da4fc703e?"}' | jq
```

## Eval and benchmarks

A committed mode × metric table lives at [`docs/benchmark.md`](docs/benchmark.md). Regenerate against a live Supabase + Voyage stack with:

```bash
python -m backend.eval.run --all-modes --benchmark
python -m backend.eval.run --all-modes --llm-judge --benchmark   # adds Claude-as-judge correctness
python -m backend.eval.run --agent-eval                          # tool selection / refusal accuracy
```

Per-mode and per-query JSON is written to `backend/eval/results/`. The browser dashboard is at `GET /eval` once the API is running.
